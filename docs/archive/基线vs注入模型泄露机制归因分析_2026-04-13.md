# 基线 vs 注入模型泄露机制归因分析（2026-04-13）

日期：2026-04-13
最后验证日期：2026-04-13
状态：主线分析文档

## 1. 结论先行

当前最值得优先解释的问题，不再是“哪一段注入层更好”，而是：

> 为什么只要启用注入，模型整体就会显著偏离 baseline 的低泄露状态。

已有证据表明：
- baseline 泄露率仅 `1.33%`；
- injected 配置普遍在 `22%` 到 `34%`；
- 即使把层位前移到当前更优的 `early_4_11`，泄露率仍有 `23.33%`；
- 因此，核心矛盾已经不是局部层位差异，而是 **baseline vs injected 的机制级差异**。

## 2. 为什么要切换分析主线

### 2.1 B1 已回答“局部最优”问题，但没解决主矛盾

B1 首轮结果：
- `early_4_11`：`2.978 / 23.33%`
- `mid_8_15`：`2.833 / 30.00%`
- `late_12_19`：`2.878 / 27.78%`

这证明：
- 当前 `[8..15]` 更差；
- 层位确实会影响结果；
- 但无论 early / mid / late，泄露率都远高于 baseline。

所以：
- B1 有价值，但已不应继续占据主优先级。

### 2.2 现在更大的差异来自“是否启用注入”

从量级看：
- baseline：`1.33%`
- injected 最好组：`23.33%`
- injected 最差组：`34.00%`

也就是说：
- 层间差异约 `6~10` 个百分点；
- baseline vs injected 差异约 `22~33` 个百分点。

这说明当前最应该优先解释的是：
- **注入这件事本身，到底把模型带到了哪里。**

## 3. 当前证据支持什么，不支持什么

### 3.1 基本已排除：训练数据普遍显式污染

训练/验证集中显式伪思考比例都约 `0.0015`，不足以解释 injected 模型中的系统性 planning 泄露。

因此当前不支持：
- “模型只是把训练集里的思考话术学出来了” 这一主假设。

### 3.2 部分支持：训练目标有价值，但提升空间有限

训练监督质量审计显示：
- gold 对 baseline 有优势；
- 但优势不压倒性。

这支持：
- 训练目标不是完全无效；
- 但它并没有大到足以抵消 injected 路径带来的副作用。

### 3.3 已基本确认：gate 不是当前主解释变量

已有 gate × leak 归因结果显示：
- leak / non-leak 的 gate 统计几乎一致；
- 不同层位实验中 gate 均值也几乎一致。

这意味着：
- 当前主问题不是“某些样本 gate 更高”；
- 更像是“注入表示本身持续把模型推向 planning/reasoning 子空间”。

### 3.4 已确认：层位会影响程度，但不是根因闭环

B1 结果说明：
- 前移能减轻；
- 中层更差；
- 但任何 8 层窗口都无法回到 baseline 状态。

这意味着：
- 层位是副作用放大器；
- 但不是主根因的完整解释。

## 4. 当前最合理的机制假设

结合已有证据，当前最合理的机制解释是：

### 假设 A：注入向量把隐藏态整体推向“先规划后回答”的区域

即使训练数据没有显式思考文本，模型内部仍可能存在：
- planning / reasoning / response drafting 的隐式中间状态。

baseline 在正常 chat-template 与 `enable_thinking=False` 下：
- 大多不会把这些中间状态外显出来。

而 injected 路径可能：
- 通过持续的隐藏态偏移，破坏原本的回答边界；
- 让“内部 planning 痕迹”更容易直接流到输出层。

### 假设 B：训练目标只奖励“更像目标回复”，但没约束“不要外显中间规划”

如果 loss 主要优化：
- token-level imitation
- persona 风格贴近

却没有明确惩罚：
- “Okay, the user is...”
- “I need to...”
- “Let me think...”

那么模型可能学到的是：
- 为了更稳地生成 persona style，先显式做规划；
- 而不是直接给出面向用户的完成态回答。

### 假设 C：baseline 与 injected 的推理行为路径并未真正对齐

当前 baseline 生成链路明确使用：
- chat template
- `/no_think`
- `enable_thinking=False`

而 injected 路径虽然不一定显式打开 thinking，但：
- 它走的是另一套生成逻辑；
- prompt 组织方式不同；
- 隐藏态又被持续注入改写；
- 这使两者并不是严格可比的“同一推理状态”。

因此：
- injected 输出泄露，不一定说明它“学会了 thinking 模板”；
- 更可能说明它已经偏离了 baseline 的稳定生成轨道。

## 5. 由此带来的优先级修正

### 第一优先：解释“为什么 injected 整体会泄露”

重点问题：
1. baseline 与 injected 在推理入口上是否真正可比；
2. injected 是否系统性生成“面向自己”的 planning 句式；
3. 这些 planning 句式是否与 persona gain 成正相关，还是纯副作用；
4. 当前 loss / 监督是否缺乏对“完成态回答”的约束。

### 第二优先：训练与推理链路对齐检查

重点检查：
- baseline 与 injected 的 prompt 结构是否一致；
- 是否都显式关闭 thinking；
- injected 的 decode 路径是否天然更容易保留内部草稿。

### 第三优先：loss 与训练目标约束

重点思路：
- 是否需要加入“禁止外显 planning”的负向约束；
- 是否需要在训练样本上增加 response-form filtering；
- 是否要把“只保留面向用户完成态回答”写入训练前处理。

### 第四优先：局部层位 / 窗口继续微调

这部分仍有价值，但已经不是主线。

## 6. 下一步最小闭环建议

建议把后续工作改成以下顺序：

### M1：baseline vs injected 推理链路对齐审计
- 对比两条生成链路的：
  - prompt 结构
  - chat template 使用方式
  - `enable_thinking` 行为
  - 解码参数

### M2：planning 外显模式审计
- 从 baseline 与 injected 各抽样 50 条；
- 统计典型 planning 开头模式；
- 比较它们的出现频次与句式类型。

### M3：训练目标与输出形式错位审计
- 检查训练 supervision 是否主要约束“内容像目标”，而没有约束“输出形式必须是完成态回答”；
- 必要时补训练前 response-form 过滤。

## 7. 一句话结论

> 当前真正该优先解释的，不是哪一段层位更好，而是为什么 PersonaSteer 只要启用注入，就会整体性地偏离 baseline 的低泄露生成轨道；层位只是这个主问题上的次级放大因素。
