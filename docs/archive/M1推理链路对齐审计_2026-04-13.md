# M1 推理链路对齐审计（2026-04-13）

日期：2026-04-13
最后验证日期：2026-04-13
状态：已完成首轮代码级审计

## 1. 结论先行

当前 baseline 与 injected 的生成链路**并没有真正对齐**。

这不是一个小差异，而是会直接影响：
- 是否使用 Qwen3 官方 chat template；
- 是否显式关闭 thinking；
- prompt 格式是否与训练一致；
- 解码输出是截取“纯新生成部分”还是“整段再切”。

因此：

> 当前 injected 与 baseline 的对比，并不是在完全相同的推理条件下进行的。仅这一点，就足以成为思考泄露差异的重要来源之一。

## 2. 审计对象

本轮主要审计以下入口：
- `scripts/v4_eval_qwen3.py`
- `src/models/persona_steer.py`
- `src/data/aloe_dataset.py`
- `src/training/trainer.py`

## 3. 发现的关键差异

## 3.1 Prompt 构造方式不一致

### baseline
在 `scripts/v4_eval_qwen3.py` 中，baseline 使用：
- system + user 双消息；
- Qwen3 `apply_chat_template(...)`；
- `add_generation_prompt=True`；
- `enable_thinking=False`；
- system 中显式加入 `/no_think`。

对应位置：
- `scripts/v4_eval_qwen3.py:174`
- `scripts/v4_eval_qwen3.py:181`

### injected
在同一脚本中，injected 使用：
- 纯字符串 prompt：`用户: {user_input}\n助手:`；
- 不走 `apply_chat_template`；
- 不传 `enable_thinking=False`；
- 不加 `/no_think` system 指令；
- `profile` 参数在生成 prompt 中没有被使用。

对应位置：
- `scripts/v4_eval_qwen3.py:199`
- `scripts/v4_eval_qwen3.py:202`

### 结论

baseline 与 injected 在输入层面根本不是同一协议：
- 一个是官方 chat-template 路径；
- 一个是手写“用户/助手”文本路径。

这意味着 injected 更容易偏离 Qwen3 原生的对话边界与输出约束。

## 3.2 Thinking 控制方式不一致

### baseline
baseline 明确做了两层抑制：
1. system prompt 中加入 `/no_think`
2. `apply_chat_template(..., enable_thinking=False)`

### injected
injected 当前没有对应控制：
- 不走 chat template；
- 没有 `/no_think`；
- 也没有等价的 thinking 关闭入口。

### 结论

即使 injected 没有“显式开启 thinking”，它也没有像 baseline 那样“显式关闭 thinking”。

换句话说：
- baseline 是**主动压制思考模式**；
- injected 更像是**裸生成 + 注入干预**。

这会显著放大两者的泄露率差异。

## 3.3 输出截取方式不完全对齐

### baseline
baseline：
- 使用 `model.generate(...)` 返回完整序列；
- 解码时只取 `outputs[0][input_ids.shape[1]:]`，即**仅新生成部分**。

对应位置：
- `scripts/v4_eval_qwen3.py:185`
- `scripts/v4_eval_qwen3.py:194`

### injected
injected：
- `PersonaSteerModel.generate(...)` 返回的是 `generated` 全序列；
- 上层直接 `decode(generated_ids[0])` 解整段；
- 再通过字符串 split `"助手:"` 截断前缀。

对应位置：
- `src/models/persona_steer.py:305`
- `scripts/v4_eval_qwen3.py:216`
- `scripts/v4_eval_qwen3.py:217`

### 结论

虽然 injected 后面做了 `split("助手:")`，但它仍不是和 baseline 一样的“仅解码新增 token”流程。

这会带来两个问题：
- 更依赖 prompt 文本结构恰好可切分；
- 一旦生成内容继续沿着 planning 句式展开，后处理更难稳定截断。

## 3.4 解码实现路径不一致

### baseline
baseline 使用 HuggingFace 原生 `backbone.generate(...)`。

### injected
injected 使用 `PersonaSteerModel.generate(...)` 自定义逐 token 循环：
- `use_cache=False`
- 手工 top-p 采样
- 手工 stop token 判定
- 固定注入向量后逐步解码

对应位置：
- `src/models/persona_steer.py:334`
- `src/models/persona_steer.py:351`
- `src/models/persona_steer.py:360`

### 结论

两者不只是 prompt 不同，连 decode runtime 都不同：
- baseline 走 backbone 原生生成栈；
- injected 走自定义生成栈。

因此，当前差异里混入了：
- 注入影响；
- 自定义解码影响；
- prompt/template 影响；
- thinking 控制差异。

## 3.5 训练与推理格式也没有完全对齐

### 训练侧
训练数据在 `src/data/aloe_dataset.py` 中明确使用：
- `apply_chat_template(...)`
- `enable_thinking=False`
- assistant 完整回复作为监督目标

对应位置：
- `src/data/aloe_dataset.py:78`
- `src/data/aloe_dataset.py:84`
- `src/data/aloe_dataset.py:92`
- `src/data/aloe_dataset.py:104`

### 推理侧 injected
当前 injected 推理并未沿用训练侧 chat-template，而是改成了：
- `用户: ...\n助手:` 的纯文本 prompt。

### 结论

这意味着 injected 当前存在一个明显的 train-infer mismatch：
- 训练时学的是 chat-template 格式；
- 推理时用的是手写纯文本格式。

这本身就可能诱发异常输出形式，包括 planning 外显。

## 4. 当前最重要的审计结论

结合以上差异，当前可以得出四个明确判断：

1. **baseline 与 injected 当前不是同等推理口径**；
2. **injected 缺失与 baseline 等价的 thinking 关闭机制**；
3. **injected 存在训练格式与推理格式不一致问题**；
4. **当前 injected 泄露率高，不能简单全归因于“注入层或注入向量本身”**，因为推理链路本身就有明显偏差。

## 5. 对后续任务的直接影响

### 应立即优先做
- 先把 injected 生成链路尽可能对齐到 baseline：
  - 同样使用 chat template；
  - 同样显式 `enable_thinking=False`；
  - 同样只 decode 新生成部分；
  - 同样保留必要的 system 指令风格。

### 在这之前不宜过度解读
- loss 是否鼓励 thinking；
- 注入层是否是唯一主因；
- 是否需要先大改训练数据。

因为如果 M1 都没对齐，后面的结论会被输入链路偏差污染。

## 6. 一句话结论

> 当前 baseline 与 injected 最大的问题之一，是两条生成链路并不等价：baseline 走官方 chat-template + 明确 no_think，而 injected 走手写 prompt + 自定义生成栈 + 无等价 thinking 关闭；在这种前提下，injected 的高泄露率很可能被推理链路差异显著放大。
