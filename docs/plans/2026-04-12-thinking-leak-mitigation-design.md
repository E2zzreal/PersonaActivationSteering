# 2026-04-12 思考泄露止损方案设计

日期：2026-04-12
最后验证日期：2026-04-12
状态：提议（已通过讨论确认）

## 1. 背景

最新权威评估报告 `docs/training_evaluation_report.md` 显示：
- baseline 分数最高；
- 注入模型存在明显思考过程泄露；
- 泄露问题已显著污染评估与对比结论。

因此，在继续扩大 PersonaSteer 训练投入之前，需要先建立一个**保守型思考泄露止损方案**。本方案的目标不是从根本上“治好”模型，而是先让评估和展示口径回到可比较、可决策状态。

## 2. 目标

本轮方案的目标是：
1. 不改训练主链路；
2. 不掩盖原始模型问题；
3. 在推理/评估侧建立一层保守型止损逻辑；
4. 同时保留原始输出与清洗后输出，支持双轨评估；
5. 帮助判断问题到底来自：
   - 外显泄露；
   - 注入机制本身；
   - 训练数据或训练目标。

## 3. 非目标

本轮不做：
- 重训模型；
- 修改 backbone 或 HyperNetwork 结构；
- 直接通过 aggressive 截断掩盖问题；
- 对训练数据做大规模清洗。

## 4. 方案选择结论

讨论中已确认采用：
- **主线策略：推理/评估侧止损优先**
- **风格：保守型**

即：
- 不进入训练路径；
- 不默认改写所有输出；
- 只在评估与展示脚本中增加轻量清洗层；
- 保留 `raw_response` 与 `clean_response` 双轨结果。

## 5. 总体架构

方案采用三层结构：

### 5.1 生成前约束
统一所有生成入口：
- 显式使用 `enable_thinking=False`（支持时）
- 对 baseline / injected 统一生成参数
- 统一解码与 prompt 模板行为

### 5.2 生成后保守止损
对模型输出执行规则检测：
- 若命中典型思考前缀，则尝试最小清理；
- 只删除明显的“面向自己”的推理片段；
- 不对正常用户回复做语义改写。

### 5.3 评估审计层
对每条样本记录：
- 原始输出
- 清洗后输出
- 是否命中泄露
- 命中了哪些模式
- 是否应用了清洗
- 是否因风险过高而跳过清洗

## 6. 数据流设计

生成链路调整为：

```text
model.generate()
  -> raw_response
  -> leak_detector
  -> conservative_cleaner
  -> clean_response
  -> 写入 responses.json（双轨字段）
```

输出字段建议至少包括：
- `raw_response`
- `clean_response`
- `leak_detected`
- `leak_patterns`
- `cleaning_applied`
- `cleaning_skipped`

评估时：
- 默认主比较使用 `clean_response`
- 同时保留 `raw_response` 的泄露统计
- 报告中必须同时呈现 raw / clean 两条口径

## 7. 规则设计

第一版只使用可审计的显式规则，不使用自由改写。

### 7.1 典型泄露模式
首批规则建议覆盖：
- `Okay,`
- `I need to`
- `the user is`
- `I should`
- `let me think`
- `my response should`

### 7.2 清理方式
采用“前缀段清理”：
- 只在文本前 1–3 句连续命中思考模式时触发；
- 优先删除前导思考段；
- 保留后续真正面向用户的回答。

### 7.3 安全回退
若清理后出现以下任一情况，则不替换正文：
- 文本过短；
- 首句残缺；
- 语义明显断裂；
- 无法判断剩余内容是否仍可作为对用户回复。

此时仅记录：
- `leak_detected=true`
- `cleaning_skipped=true`

## 8. 成功标准

本轮成功标准不是“清洗后一定超过 baseline”，而是：

1. 能稳定识别并量化思考泄露；
2. 能在不明显伤害自然度的前提下降低泄露率；
3. 能输出 raw / clean 双轨结果；
4. 能支持判断：问题主要在外显泄露，还是更深层机制。

## 9. 验证策略

采用双轨验证：

### 9.1 指标
每个配置同时统计：
- 原始泄露率
- 清洗后泄露率
- 原始 Judge 分数
- 清洗后 Judge 分数
- 被清洗样本比例

### 9.2 结果解释
若出现以下情况：

#### 情况 A：清洗后分数明显回升
说明外显泄露是主问题，下一步可优先优化生成口径。

#### 情况 B：清洗后仍不如 baseline
说明问题更深，可能在注入机制、训练目标或数据分布。

#### 情况 C：清洗后分数下降
说明规则误伤正常输出，应收缩清洗策略。

## 10. 实施范围

建议优先接入以下脚本：
- `scripts/generate_all_conversations.py`
- `scripts/judge_v3_multidim.py`
- `scripts/eval_qwen3.py`
- `scripts/eval_qwen3_baseline.py`
- 其他当前主评估入口

## 11. 推荐实施顺序

1. 抽出公共 `leak_detector / conservative_cleaner` 工具；
2. 在主评估脚本中接入双轨输出；
3. 先对 baseline 与 `stage3_v2` 跑一轮双轨评估；
4. 根据结果决定：
   - 是否继续做训练数据清洗；
   - 是否需要调整注入机制。

## 12. 风险与边界

### 风险一：误伤正常回复
缓解：
- 只处理前缀段；
- 命中规则后仍保留原始输出；
- 不做自由改写。

### 风险二：用清洗掩盖模型缺陷
缓解：
- 所有报告同时保留 raw / clean 结果；
- baseline 门槛同时参考 raw 泄露率。

### 风险三：不同脚本口径再次分叉
缓解：
- 接入统一公共函数；
- 结果文件强制写入相同字段。

## 13. 一句话结论

> 当前“思考泄露止损方案”应以推理/评估侧的保守型双轨清洗为主：既不污染训练，也不掩盖问题，而是先把泄露量化、隔离并纳入统一评估，为后续判断是否要动训练数据预处理或注入机制提供可靠依据。
