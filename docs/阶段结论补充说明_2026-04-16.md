# 阶段结论补充说明（2026-04-16）

日期：2026-04-16
最后验证日期：2026-04-16
状态：补充说明

## 1. 目的

本文用于补充近期在讨论中已经达成、但尚未系统沉淀到文档中的几类判断：
- 为什么当前主问题已从“止损 thinking 泄露”转为“注入增益不足”；
- 为什么下一阶段应优先关注数据集上限、评估敏感度、loss 目标与注入结构；
- 为什么“大模型造数据”必须先经过教师能力审计，而不能直接大规模蒸馏。

## 2. 当前主问题的重新定义

在统一推理链路完成后，当前项目主问题已发生变化：

### 旧问题
- injected 模型是否系统性高泄露；
- 是否需要优先做止损与清洗。

### 新问题
- injected 模型的副作用已大幅下降；
- 但 injected 相对 baseline 的收益提升幅度仍不够大；
- 因此当前主问题是：

> 为什么在统一链路后，PersonaSteer 的人格化增益仍然偏小。

## 3. 当前最值得优先解释的四类原因

### 3.1 数据集监督上限可能不足

已有 50 条 `gold vs baseline` 审计结果显示：
- gold 对 baseline 有优势；
- 但优势不压倒性；
- 排除 error 后有效样本胜率约 `64.29%`。

这意味着：
- 当前训练目标不是无价值；
- 但它的上限可能并不高；
- 模型很可能只是学到了“略优于 baseline 的目标”，而非显著优于 baseline 的目标。

因此，数据层是当前最值得优先排查的上限瓶颈之一。

### 3.2 评估口径可能低估 persona 差异

统一链路后的权威结果显示：
- stage2、baseline、stage1、stage3 的分数差距很小；
- 当前 judge prompt 可能更偏向：
  - 自然度
  - 流畅度
  - 通用高质量回答

而不一定足够敏感于：
- 人格特质是否鲜明；
- 不同 persona 的边界是否明显；
- 回答是否真正体现目标 persona 的价值排序与表达偏好。

因此，评估层可能低估了真实 persona 增益，也可能掩盖了数据集/模型的差异。

### 3.3 当前 loss 更偏保守保底，而不是拉高 persona 增益

当前训练主干仍以 SFT 为主，dual loss 主要作用更接近：
- 保证 injected 不比 clean 差太多；
- 惩罚注入导致的质量下降。

这类目标更偏向：
- “不要变差”

而不是：
- “更像目标人格，且差异更鲜明”。

因此，当前 loss 设计可能天然更容易让模型收缩到安全但有限提升的区域，而不是主动拉开 persona 增益。

### 3.4 当前注入方式可能表达力不足

虽然这不是当前第一优先解释变量，但仍值得保留：
- 当前 layer-level 注入可能过于粗粒度；
- 即使定位到较优阶段，也未必真正打到最关键的 persona 表达子空间；
- 后续若数据与评估层都确认存在更高上限，注入结构（层位、层数、注意力头级别）仍应作为后续技术路线保留。

## 4. 当前优先级排序建议

基于现有证据，当前更合理的归因优先级是：

1. **数据层（D）**：训练监督上限是否太低
2. **评估层（E）**：评估是否低估 persona 差异
3. **损失层（L）**：loss 是否只保底、不拉高 persona 增益
4. **注入层（I）**：结构表达力是否不足

简写为：
- `D > E > L > I`

## 5. 关于“大模型造数据”的最新共识

近期讨论已经形成一个明确共识：

### 5.1 这本质上是一种广义蒸馏

如果使用更强模型：
- 生成候选回答；
- 评审 preferred / rejected；
- 提供失败标签与修订意见；

那么它本质上是在蒸馏：
- 偏好信号；
- 完成态回答边界；
- persona 一致性的判断标准。

### 5.2 不能默认强模型天然具备“更好人格化能力”

强模型往往具备：
- 更自然；
- 更稳定；
- 更礼貌；
- 更少低级错误。

但这并不等价于：
- 更强 persona 区分能力。

如果直接蒸馏其默认输出，很容易发生：
- 统计平均坍缩；
- 回答越来越自然，却越来越中性；
- 不同 persona 之间差异被高质量平均文风抹平。

### 5.3 正确做法不是直接蒸馏“默认输出”

而是先验证教师模型是否具备：
- 人格区分能力；
- 完成态回答能力；
- 低 thinking 泄露；
- 低平均化倾向。

通过后，才蒸馏其：
- 受控 persona 偏好；
- preferred / rejected 对；
- failure tags；
- 完成态回答行为。

## 6. 关于双模型路线的当前判断

近期已形成如下共识：

### 6.1 第一阶段更适合“双模型离线造数”

即：
- 一个强模型做候选生成器；
- 一个强模型做评审器；
- 先构建偏好数据与失败标签。

这比立即训练 reward model / 辨别器更合适，因为：
- 启动更快；
- 风险更低；
- 可以先验证数据上限是否足够高。

### 6.2 辨别器/奖励模型应作为第二阶段

训练一个辅助辨别模型当然值得做，但更适合在：
- 已有一批高质量偏好数据后，
- 再进行蒸馏与压缩。

否则容易发生：
- 轻量 judge 只学到表面风格偏见；
- 没学到真正的人格区分标准。

因此当前更推荐：
- 先双模型造数；
- 再蒸馏轻量辨别器；
- 再考虑接入生成闭环。

## 7. 关于 Claude-Opus-4.6 作为生成器的补充判断

讨论中已形成如下结论：
- `Claude-Opus-4.6` 适合作为**第一轮高质量候选生成器**；
- 但不能先验假设它就是“合格 persona teacher”；
- 它更适合被定位为：
  - 候选生成器，
  - 而不是直接提供单一真值答案的标准教师。

其潜在风险在于：
- 自然度很高；
- 指令遵循较强；
- 但人格差异可能被统一为一种高级平均风格。

因此才需要先做“教师人格区分能力审计”。

## 8. 当前阶段建议

综合以上补充判断，当前更合理的下一阶段工作顺序是：

1. 运行教师人格区分能力审计；
2. 判断 Claude-Opus-4.6 是否具备足够 persona 区分能力；
3. 若通过，再进入小规模 preferred / rejected 造数；
4. 若不通过，优先迭代：
   - persona 文案；
   - generator prompt；
   - judge rubric；
   - 模型组合。

## 9. 一句话结论

> 当前 PersonaSteer 的主瓶颈已经从“高泄露止损”转向“有效增益不足”；最值得优先排查的是训练监督上限、评估敏感度以及教师模型是否真的具备 persona 区分能力，而不是直接把大模型默认输出大规模蒸馏进训练集。
