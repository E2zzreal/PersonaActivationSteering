# 教师人格审计与三源评估设计（2026-04-16）

日期：2026-04-16
最后验证日期：2026-04-16
状态：已完成脚本实现，生成阶段待执行

## 1. 本次工作目标

1. 审计 Claude-Opus-4.6 作为教师模型是否具备 persona 区分能力；
2. 修复 persona 文案与 generator prompt，使三个 persona 产生可测量结构性差异；
3. 建立三源（ALOE gold / Claude-Opus-4.6 / Qwen3-4B baseline）统一打分管道；
4. 将评估升级为多轮、多模型、严格 rubric 口径。

---

## 2. 教师人格审计 v1 结论（已完成）

### 2.1 实验设置

- 生成模型：Claude-Opus-4.6
- 评审模型：GPT-5.2
- 样本：20 user_input × 3 persona × 2 候选 = 120 条
- 评审对：60 对（同 persona 双候选 A/B 比较）

### 2.2 v1 结果

| 指标 | 结果 |
|------|---:|
| 人格差异强 (difference_strong) | **11.7%** |
| 平均化坍缩 (avg_collapse) | **40.0%** |
| 泄露对 (leak_pair) | 11.7% |

各 persona 细分：
- p1_warm_extraverted: strong=10%, collapse=45%
- p2_calm_detail_oriented: strong=**0%**, collapse=35%
- p3_rational_goal_oriented: strong=25%, collapse=40%

### 2.3 v1 根因

Claude-Opus-4.6 天然具备高质量、高流畅度，但会将不同 persona 统一为一种"高质量平均风格"。v1 persona 文案太抽象，约束力不足，无法防止坍缩。

## 3. 教师人格审计 v2（已完成）

### 3.1 改进内容

**强化 persona 文案**：为三个 persona 各设置强制结构合同（REQUIRED / FORBIDDEN）：

| | p1_warm | p2_calm | p3_rational |
|--|--|--|--|
| emoji | 必须≥1 | 禁止 | 禁止 |
| 感叹号 | 必须≥2 | 禁止 | ≤1 |
| 追问数 | 必须≥2 | 恰好1个（精确） | 恰好1个（聚焦行动） |
| 开头风格 | Oh wow! / I love that! | 平静承接 | 直接切入判断 |

**强化 generator prompt**：加入"可辨别性测试"——如果这条回复换个 persona 也说得通，视为失败，需重写。

**配置文件**：`configs/audit/teacher_persona_audit_v2.json`（含 `generator_prompt` 字段，可在 config 中覆盖默认 prompt）

## 4. 三源统一 LLM Judge 打分（v2 单轮，已完成）

### 4.1 打分口径

- 评分模型：GPT-5.2（单模型，v2 结果）
- Prompt：简单 1-5 分，无行为锚点（当前已知为弱口径，见第 5 节）
- 评估方式：单轮，Claude 用 p1/p2/p3 persona，ALOE/Qwen3 用原始 personality

### 4.2 v2 结果

| 来源 | n | 均分 |
|------|--:|---:|
| Claude-Opus-4.6 v2 | 120 | **4.625** |
| Qwen3-4B baseline | 90 | 3.711 |
| ALOE gold | 120 | **3.217** |

### 4.3 关键发现

> ALOE gold 的人格对齐分（3.217）**低于 Qwen3 baseline（3.711）**。

这意味着：当前训练数据的人格一致性，不仅没有超过模型自生成的 baseline，甚至还不如它。这直接解释了 PersonaSteer 注入后增益有限的根本原因——模型在学一个比自己默认行为更弱的人格信号。

归因优先级 `D > E > L > I` 中的 D 层（数据上限）瓶颈已被量化。

## 5. 评估体系升级（已实现，待执行）

### 5.1 问题清单

旧版 judge（`score_three_sources.py` v1）的四个缺陷：
1. 无 user_input 上下文
2. 评分锚点模糊，"好回复"容易得高分
3. 单维度，无法区分风格/内容/一致性
4. temperature=1.0 单次采样，方差高

### 5.2 新版 judge 设计

**Prompt**：采用 `judge_v1_strict_rubric.py` 的严格 rubric 逻辑：
- 每个分值有行为锚点描述
- 明确保护规则："好但通用 = 3分，5分只给 persona 专属性"
- CoT：先写 1-2 句分析，再输出 `Score: X`
- temperature 降至 0.2

**多模型集成**：GPT-5.4 + GLM-5.1 两模型各自打分取均值，在 `configs/api_config.yaml` 的 `judge_models` 字段配置。

### 5.3 多轮对话评估管道

**核心设计目标**：三源评估同一批对话，保证可比性。

**生成脚本**：`scripts/audit/generate_parallel_dialogues.py`
- 从 ALOE val 采样 N 条对话（默认 30 条，每条前 5 轮）
- 所有三源使用**相同 ALOE personality**（统一格式）
- ALOE：直接取 gold 回复
- Claude：API 逐轮生成，携带自身历史（自生成轨迹）
- Qwen3：本地模型逐轮生成，携带自身历史（自生成轨迹）
- 支持断点续传、超时重试

**打分脚本**：`scripts/audit/score_three_sources.py`（已重构）
- 输入：`generate_parallel_dialogues.py` 的输出（并行对话 JSON）
- 三路全部走多轮 judge（`MULTITURN_PROMPT`）
- 断点续传

**运行顺序**：
```bash
# 第一步：生成并行对话
python scripts/audit/generate_parallel_dialogues.py \
  --output results/parallel_dialogues/dialogues.json \
  --n 30 --max_turns 5 --device cuda:0

# 第二步：三源打分
python scripts/audit/score_three_sources.py \
  --parallel_input results/parallel_dialogues/dialogues.json \
  --output_dir results/three_source_score_v3
```

## 6. 脚本健壮性改进

所有 audit 脚本（`run_teacher_persona_audit.py`、`judge_teacher_persona_audit.py`、`score_three_sources.py`）统一加入：
- **超时**：`timeout` 参数，默认 60-90s
- **指数退避重试**：失败后 2/4/8s 重试，最多 3 次
- **断点续传**：启动时加载已有结果，跳过已完成项
- **API 配置 fallback**：优先读环境变量，fallback 到 `configs/api_config.yaml`
