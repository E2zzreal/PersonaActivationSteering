# P0 修复方案设计文档

日期：2026-04-12
最后验证日期：2026-04-12
状态：提议
范围：PersonaSteer 当前主线训练与评估体系的 P0 级问题修复方案

## 1. 文档目的

本文用于定义 PersonaSteer 当前最优先修复的 P0 问题，并给出一套**最小闭环、可落地、可验证**的修复设计方案。目标不是一次性重构整个项目，而是先打通“训练配置 → 模型生效 → 训练目标有效 → 结果可验证”这条主链路。

## 2. 背景

根据当前仓库代码、配置和历史文档，项目虽然已有完整训练与评估框架，但存在若干会直接影响实验结论可信度的关键问题：

1. gate 关键超参未真正从配置注入模型；
2. Stage 2 / Stage 3 的对比学习在当前 batch 设置下基本失效；
3. 主训练目标偏 SFT，无法强约束多轮人格一致性；
4. 主训练与主评估路径存在配置分叉，导致实验复现与归因困难。

这些问题不解决，即使局部结果看起来“还行”，也难以证明 PersonaSteer 的方法本身有效。

## 3. P0 范围定义

本次 P0 只覆盖下面三类问题：

### P0-1：gate 配置未生效
问题表现：
- `gate_init_bias`、`gate_max` 写在 YAML 中，但未真正传给 `SteeringInjection` / `DynamicGate`
- 导致实验配置与真实训练行为不一致

### P0-2：对比学习训练链路失效
问题表现：
- Stage 2 / Stage 3 虽配置了 `scl_weight`
- 但常用训练配置的 `batch_size=1`
- `SupervisedContrastiveLoss` 在单样本时直接返回 0

### P0-3：主线配置与结果归因混乱
问题表现：
- 同类 stage 存在多个配置和变体目录
- 文档、脚本、checkpoint 命名不完全收敛
- 难以明确哪个实验是真正“主线结果”

   当前 `src/data/aloe_dataset.py:90` 已加入 `enable_thinking=False`，与生成侧行为更一致。这减少了训练数据模板与推理模板错位带来的噪声，因此该项从 P0 降为已修复事项，文档中只保留验证要求。

   `3cbf153` 新增并行训练和评估脚本，提高了实验吞吐，但不会自动修正“实验是否有效”的根问题。若核心配置未生效、对比损失无效，那么并行化只会更快地产出不可靠结果。

结论：近期脚本修正说明项目在向正确方向收敛，但**原 P0 修复方案不需要推翻，只需要补充“已修项”和“优先级微调说明”**。

## 4. 非目标

本次 P0 **不**处理以下事项：

1. 不做大规模架构重写；
2. 不引入 head-level 注入等高复杂度新结构；
3. 不重做整套数据标注体系；
4. 不追求一次性把最终个性化效果拉到最佳；
5. 不清理全部历史实验资产。

本次目标只有一个：

> 让当前主方法的关键训练假设真正成立，并能被可信地验证。

## 5. 成功标准

P0 修复完成后，应满足以下最低标准：

### 5.1 配置生效性
- 修改 YAML 中 `gate_init_bias` 后，模型初始化时对应 bias 实际变化；
- 修改 YAML 中 `gate_max` 后，训练与推理 gate 上界实际变化；
- 日志中明确打印实际 gate 配置值。

### 5.2 对比学习有效性
- Stage 2 / Stage 3 的 `scl_loss` 不再长期恒为 0；
- 至少部分 batch 中存在正样本对；
- 训练日志中能观测到 `sft_loss` 与 `scl_loss` 的分离变化。

### 5.3 主线可复现性
- 明确 1 套官方主线配置（Stage 1 / 2 / 3 各 1 份）；
- 明确 1 套官方训练脚本调用方式；
- 明确 1 套官方评估入口；
- 结果目录与日志目录命名统一。

### 5.4 目标验证性
- baseline 与修复后主线结果可以在同一评估协议下进行比较；
- 能判断“修复后方法是否比 baseline 更强”，而不是仅看到训练 loss 变化。

## 6. 总体修复策略

采用“**先打通配置与训练有效性，再统一主线，再观察结果**”的顺序。

### 修复原则
1. 最小改动边界；
2. 不引入不必要新抽象；
3. 优先修复会破坏实验有效性的缺陷；
4. 每一步都能独立验证；
5. 修复后必须能形成标准主线。

## 7. 方案设计

## 7.1 P0-1：gate 配置贯通修复

### 7.1.1 问题根因
当前 `DynamicGate` 支持配置化参数，但 `SteeringInjection` 构造 gate 时直接硬编码：
- `gate_max=1.0`
- `gate_init_bias=-2.0`

这导致训练配置与模型行为脱节。

### 7.1.2 设计目标
让以下参数从配置文件贯穿到实际模型初始化：
- `gate_max`
- `gate_init_bias`
- 后续可预留 `gate_dropout`、`gate_hidden_dim` 的一致传递

### 7.1.3 设计方案

#### 方案 A：扩展 `PersonaSteerConfig`
在 `PersonaSteerConfig` 中新增字段：
- `gate_max`
- `gate_init_bias`

由 `scripts/train.py` 的 `create_model()` 负责从 YAML 读取并写入配置对象。

#### 方案 B：扩展 `SteeringInjection.__init__`
新增构造参数：
- `gate_max: float = 1.0`
- `gate_init_bias: float = -2.0`

并在内部传给 `DynamicGate`。

#### 方案 C：模型初始化日志显式打印
在模型创建阶段打印：
- inject_layers
- gate_hidden_dim
- gate_max
- gate_init_bias

确保实验日志可以审计。

### 7.1.4 推荐实现
采用 **A + B + C**。

理由：
- 配置链路完整；
- 变更局部；
- 便于后续扩展；
- 可直接验证。

### 7.1.5 验收方法
1. 将配置中的 `gate_init_bias` 改为 `-0.5`；
2. 启动模型并打印最后一层 gate bias；
3. 核对日志与参数值一致；
4. 对同一输入比较修复前后 gate 分布均值，应出现明显变化。

## 7.2 P0-2：对比学习有效化修复

### 7.2.1 问题根因
`SupervisedContrastiveLoss` 依赖 batch 内存在正样本对，但当前配置大量使用 `batch_size=1`。这会让 Stage 2 / 3 的 SCL 设计名义存在、实际失效。

### 7.2.2 设计目标
让对比学习从“代码路径存在”变成“训练中真正产生梯度信号”。

### 7.2.3 可选方案比较

#### 方案 A：直接把 batch_size 提高到 4 或 8
优点：
- 最简单；
- 不改损失函数主逻辑；
- 直接让 SCL 有正样本机会。

缺点：
- 显存压力上升；
- 若数据采样无组织，仍可能没有正样本。

#### 方案 B：实现按 user/personality 分组采样的 batch sampler
优点：
- 更可靠地产生正样本对；
- 对 SCL 最友好；
- 可在较小 batch 下仍有效。

缺点：
- 需要改 dataloader 采样逻辑；
- 工程复杂度中等。

#### 方案 C：保留物理 batch 小、做 gradient accumulation，但构造“逻辑对比批次缓存”
优点：
- 显存更友好；
- 可以跨多个 step 聚合对比样本。

缺点：
- 实现复杂；
- 与当前代码差距大；
- 容易引入状态管理问题。

### 7.2.4 推荐实现
采用 **A + B 的组合策略**：

1. Stage 2 / Stage 3 主线训练配置中，将 `batch_size` 调整到至少 `4`；
2. 新增一个最小版本的“同 user_id / 同 personality 优先成批”采样策略；
3. 若显存不足，再退化为 `batch_size=2 + gradient_accumulation`，但仍要求同批能形成正样本。

### 7.2.5 关键实现要点

#### 数据侧
- 统计 `user_id`、`personality` 分布；
- 构建 `group -> sample indices` 映射；
- 训练时优先从同组取多个样本进入同一 batch。

#### 损失侧
- 训练日志中显式打印：
  - `scl_loss`
  - `num_positive_pairs`
  - `batch_positive_ratio`
- 若某 batch 无正样本，记录但不作为常态。

#### 配置侧
新增字段：
- `gradient_accumulation_steps`
- `min_positive_pairs_per_batch`
- `contrastive_group_by: user_id | personality`

### 7.2.6 验收方法
1. 在 Stage 2/3 训练日志中记录 `scl_loss`；
2. 连续多个 epoch 检查 `scl_loss` 非零比例；
3. 抽样检查 batch 中确实含有正样本对；
4. 确认 `scl_loss` 对总 loss 有实质贡献。

## 7.3 P0-3：主线配置收敛修复

### 7.3.1 问题根因
当前项目实验变体过多，训练配置、checkpoint 路径、脚本入口没有完全收敛到单一主线。

### 7.3.2 设计目标
建立一个“官方主线”，后续所有效果结论默认以此为准。

### 7.3.3 设计方案

#### 配置层统一
仅保留或明确主推下列配置：
- `configs/train_stage1_main.yaml`
- `configs/train_stage2_main.yaml`
- `configs/train_stage3_main.yaml`
- `configs/eval_main.yaml`

旧配置不删除，但标记为：
- archived
- ablation
- historical

#### 输出层统一
主线输出目录统一为：
- `checkpoints/main/stage1/`
- `checkpoints/main/stage2/`
- `checkpoints/main/stage3/`
- `results/main/...`
- `logs/main/...`

#### 脚本层统一
统一训练命令：
- Stage 1：`python scripts/train.py --config configs/train_stage1_main.yaml`
- Stage 2：`python scripts/train.py --config configs/train_stage2_main.yaml`
- Stage 3：`python scripts/train.py --config configs/train_stage3_main.yaml`

统一评估命令：
- `python scripts/evaluate.py --config configs/eval_main.yaml --checkpoint ...`

### 7.3.4 验收方法
1. 新成员只看主线配置即可启动实验；
2. 日志、checkpoint、results 可一一对应；
3. baseline 与 stage 结果可以在统一协议下横向比较。

## 8. 实施顺序

### 第 1 步：修 gate 配置链路
原因：这是最小、最明确、最会污染实验结论的缺陷。

### 第 2 步：修 Stage 2/3 的 contrastive batch 机制
原因：这是当前训练目标失效的核心问题。

### 第 3 步：统一主线配置与输出路径
原因：若不先统一主线，后续所有修复结果仍难归因。

### 第 4 步：重新跑一轮最小主线验证
建议只跑：
- baseline
- stage1_main
- stage2_main
- stage3_main

并在同一评估协议下比较。

## 9. 验证计划

## 9.1 单元级验证
- 验证 `SteeringInjection` 初始化参数是否与 YAML 一致；
- 验证不同 `gate_init_bias` 下初始 gate 分布是否变化；
- 验证 contrastive sampler 是否能构造正样本 batch。

## 9.2 集成级验证
- 跑 Stage 2 / Stage 3 小规模训练；
- 观察 `sft_loss`、`scl_loss`、gate mean/std；
- 确认 loss 曲线与日志字段正常。

## 9.3 效果级验证
统一比较：
- baseline
- 修复前主线
- 修复后主线

重点看：
- LLM Judge 平均分
- 跨轮人格一致性
- 空回复比例
- gate 分布是否合理

## 10. 风险与缓解

### 风险一：batch 增大带来显存不足
缓解：
- 使用 `gradient_accumulation_steps`
- 优先降低 `max_turns`
- 保证逻辑 batch 仍有正样本

### 风险二：修复后 gate 过强导致生成质量下降
缓解：
- 保留 `gate_max`
- 对 `gate_init_bias` 做小范围网格搜索
- 记录 gate 分布与空回复率

### 风险三：对比学习生效后训练不稳定
缓解：
- 先用较小 `scl_weight`
- 监控 `scl_loss` 与总 loss 比例
- 必要时分阶段增加对比损失权重

### 风险四：主线统一后暴露旧结果不可复现
缓解：
- 接受“旧结果仅供参考”；
- 从修复后的主线重新建立基线。

## 12. 推荐落地策略

如果只允许做最小闭环，我建议严格按以下顺序：

1. **先修 gate 配置生效**
2. **再让 Stage 2/3 的 SCL 真正有信号**
3. **再统一主线配置和目录**
4. **最后重新跑最小验证闭环**

不要一上来同时改：
- 数据格式
- 注入结构
- 损失体系
- backbone 微调策略

否则你将很难知道提升来自哪里，也很难知道失败来自哪里。

## 13. 一句话结论

> PersonaSteer 当前最紧急的 P0 任务，不是继续做新实验，而是先修复“配置未生效、对比学习无效、主线不收敛”这三条会直接破坏结论可信度的问题；只有把这三条打通，后续关于个性化提升的讨论才有统计和工程上的意义。


## 14. 结合当前脚本运行状况的方案修订结论

### 14.1 不需要推翻原方案
当前最近提交已经修复了两个重要但次一级的问题：
- 阶段配置时误解冻 encoder；
- Qwen3 thinking 模式导致的训练/推理模板不一致。

这说明原方案中的部分隐含风险已经被降低，但**不会改变 P0 主矛盾**。

### 14.2 需要调整的地方

#### 调整一：把“冻结逻辑检查”从 P0 主问题改为回归检查项
后续实施时，只需确保：
- Stage 1：gate 冻结，encoder 不解冻；
- Stage 2/3：gate 解冻，encoder 仍保持冻结；
- 训练日志输出可训练参数摘要。

#### 调整二：把“thinking 模板一致性”改为已修复前提
后续若做 P0 实施，无需再把这件事作为核心修复目标，但建议保留单元测试与回归测试。

#### 调整三：保留并强化 P0-1、P0-2
结合当前脚本现状，最关键的实施顺序建议更新为：
1. 修复 gate 超参配置贯通；
2. 修复 Stage 2/3 contrastive batch 设计；
3. 统一主线配置和输出路径；
4. 在并行训练流水线中固化主线运行入口。

### 14.3 对当前运行状况的判断
如果按照当前仓库继续直接跑更多实验：
- **训练能跑**，不代表**结论可信**；
- **流水线更完善**，不代表**核心训练假设已生效**；
- **脚本修正增多**，说明项目在变好，但还没触达最关键的效能瓶颈。

因此，建议继续保留本方案作为当前主修复蓝图，只在实施说明中明确：
- 已修复项：encoder 冻结问题、thinking 模式问题；
- 未修复核心项：gate 配置贯通、对比学习有效化、主线收敛。
