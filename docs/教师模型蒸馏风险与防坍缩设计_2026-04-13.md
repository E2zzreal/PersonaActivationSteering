# 教师模型蒸馏风险与防坍缩设计（2026-04-13）

日期：2026-04-13
最后验证日期：2026-04-13
状态：方案设计

## 1. 背景

在统一推理链路后，当前 PersonaSteer 的主问题已从“系统性思考泄露”转为“注入增益不足”。当前小样本权威评估显示：
- baseline：`3.556 / 1.11%`
- stage2：`3.589 / 2.22%`
- 其他阶段与 baseline 的差距也较小。

这说明：
- 模型并非完全无效；
- 但训练收益仍偏低；
- 一个高概率原因是：训练监督目标本身的上限不够高。

因此，下一步自然会考虑：
- 用更强 LLM 生成更高质量的人格化监督数据；
- 或引入外部心理学数据集增强人格表征。

但这会带来一个核心风险：

> 如果直接蒸馏“大模型默认输出”，很可能把学生模型训练成更自然、更平滑、但人格差异更弱的“统计平均聊天模型”。

## 2. 核心问题

本设计要回答的不是：
- “能不能用大模型造数据？”

而是：
- “如何在使用大模型时，避免人格化信号被平均化、平滑化、模板化，从而发生统计平均坍缩？”

## 3. 统计平均坍缩是什么意思

这里的“坍缩”指的是：
- 模型输出越来越自然、稳定、礼貌；
- 但不同 persona 之间的表达差异越来越弱；
- 最终学到的是“高质量通用聊天风格”，而不是“更鲜明的人格化表达”。

典型表现包括：
1. 不同 persona 下回答越来越像；
2. 情绪和价值排序差异变弱；
3. 风格特征被高情商、通用式表达抹平；
4. 模型更会说话了，但“更像谁”这件事并不明显。

## 4. 为什么直接用强模型造数据会坍缩

强模型通常具备：
- 更高流畅度；
- 更强常识；
- 更稳定礼貌；
- 更少明显错误。

但这些优点并不等于：
- 更强的人格区分能力。

如果只要求它“生成更好的回答”，很容易得到：
- 自然、通顺、正确，
- 但高度中性、平均化、缺乏鲜明人格差异的输出。

因此，不能蒸馏：
- 教师模型的默认风格，

而应蒸馏：
- **教师在明确人格约束下的受控行为**。

## 5. 正确蒸馏的对象是什么

建议蒸馏以下内容，而不是单纯的答案文本：

1. **人格偏好信号**
   - 哪个回答更符合目标人格
2. **完成态回答边界**
   - 只保留面向用户的最终回答，不保留草稿式 planning
3. **负例对照**
   - 哪些回答虽然自然，但人格弱、偏移或平均化
4. **失败标签**
   - 如：thinking 泄露、persona 不鲜明、语气不匹配、过度泛化

## 6. 防坍缩的四条核心原则

### 6.1 生成正例时必须显式要求“人格差异可辨”

不能只要求：
- 更自然
- 更流畅
- 更好

必须同时要求：
- 更符合目标人格；
- 与中性回答相比存在可辨差异；
- 差异体现在语气、关注点、价值排序、信息取舍，而不是浮于表面的形容词。

### 6.2 必须同时保留反例

只蒸馏正例会把边界学丢。

建议同时构造：
- preferred response
- rejected response

rejected 不一定是错误回答，很多时候更有价值的是：
- 自然但人格弱
- 自然但过于平均化
- 自然但 persona 偏移
- 含轻微 thinking 痕迹

### 6.3 必须审查“不同 persona 是否真的区分开”

教师模型是否合格，不能看单条质量，而要看：
- 同一 user_input 下，不同 persona 的回答是否真有稳定差异。

如果教师在不同 persona 下输出高度相似，说明它不适合作为 persona teacher。

### 6.4 必须限制修订模型的重写自由度

如果采用“生成器 + 评审器 + 修订器”链路，修订器不能无限制重写。否则最后学到的只是修订器自己的平均风格。

更合适的做法是：
- 先排序；
- 再小幅修订；
- 保留原候选、评审意见、修订版三者。

## 7. 推荐的数据生产路线

建议采用三阶段路线：

### 阶段 A：教师能力审计
先不急着大规模造数据，先确认教师是否真有人格区分能力。

### 阶段 B：双模型偏好数据生产
- 生成器：产出多候选
- 评审器：排序、打标签、必要时小修

### 阶段 C：再训练学生模型/辨别器
- SFT / DPO / reranker / reward model

## 8. 最小实验方案：教师人格区分能力审计

## 8.1 目标

验证候选教师模型是否具备：
- 在相同 user_input 下稳定生成**可区分的人格化回答**；
- 同时保持自然、完成态、低 thinking 泄露。

## 8.2 实验问题

给定同一个 user_input，教师模型在不同 persona 下生成的回答：
1. 是否自然；
2. 是否没有 thinking 泄露；
3. 是否彼此可区分；
4. 是否能被评审模型稳定识别出“更符合对应 persona”。

## 8.3 最小数据规模

建议首轮：
- `20` 个 user_input
- `3` 个对比 persona
- 每个 persona 生成 `2` 个候选

总生成量：
- `20 × 3 × 2 = 120` 条候选

这已经足够看出：
- 教师是否有 persona 区分能力；
- 是否严重平均化。

## 8.4 persona 选择原则

不要选太接近的人格；首轮建议选差异大的三组：
- `外向 / 高社交 / 乐观`
- `内向 / 克制 / 细致`
- `理性 / 分析型 / 目标导向`

这样更容易验证教师是否真的拉开风格差异。

## 8.5 样本格式

每条样本至少包含：
- `sample_id`
- `user_input`
- `persona_id`
- `persona_text`
- `candidate_a`
- `candidate_b`
- `leak_flag_a`
- `leak_flag_b`
- `judge_preference`
- `judge_difference_strength`
- `failure_tags`

## 8.6 审计指标

建议至少统计：

### 指标 1：泄露率
- 候选中命中 thinking-like 模式的比例

### 指标 2：人格可辨率
- 评审模型认为“该回答明显符合目标 persona”的比例

### 指标 3：人格区分强度
- 在同一 user_input 下，不同 persona 的回答被判定“差异明显”的比例

### 指标 4：平均化率
- 被判定为“自然但过于中性/平均化”的比例

## 8.7 合格标准（首轮）

首轮可采用保守门槛：
- 泄露率 `< 5%`
- 人格可辨率 `> 60%`
- 人格区分强度 `> 50%`
- 平均化率 `< 30%`

若达不到，说明教师当前 prompt 或模型选择不适合直接造 persona 数据。

## 8.8 实验结论如何使用

### 若教师表现好
则进入下一阶段：
- 生成 preferred / rejected 对；
- 构建高质量偏好数据集。

### 若教师表现差
则优先改：
- prompt 设计；
- persona 表述方式；
- 教师模型组合；
- 评审 rubric。

而不是急着大规模造数据。

## 9. 当前建议

当前最推荐的下一步不是直接开大规模蒸馏，而是：

1. **先做教师人格区分能力审计**；
2. 验证教师模型是否真能稳定表达目标 persona；
3. 再决定是否值得做大规模 LLM 合成数据；
4. 若通过，再进入双模型偏好数据生产。

## 10. 一句话结论

> 大模型造数据本身并不自动等于更强人格化监督；如果不加控制，它更可能把学生模型带向“自然但平均化”的统计平均坍缩。正确做法不是直接蒸馏教师默认输出，而是先验证教师是否具备稳定的人格区分能力，再蒸馏其受控的人格偏好与完成态回答行为。
