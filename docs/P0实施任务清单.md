## 1. 文档目的

本文将 P0 修复方案拆解为可以直接执行、分配、验收的任务清单。与首版不同，本版明确将“思考泄露止损”和“baseline 门槛判定”前置，避免继续为负收益注入模型投入训练资源。

## 2. 实施总原则

1. 先修会破坏实验有效性的缺陷；
2. 先控制副作用，再继续扩大训练；
3. 每项任务必须有明确输入、输出、验收标准；
4. 每项任务尽量控制在最小改动边界；
5. 优先保证“实验结论可信”，其次才是“实验跑得更快”；
6. 若注入模型未超过 baseline，默认不进入下一轮扩大实验；
7. 已修复项只做回归验证，不重复大改。

## 4. 任务清单

## T0：建立思考泄露监控与止损机制

### 目标
先把注入模型最明显的负面副作用量化并纳入止损控制，避免训练/评估结论被“泄露的思考文本”污染。

### 问题背景
最新评估报告显示：
- `baseline` 思考泄露率约 `1.3%`
- `stage3_v2` 思考泄露率约 `34.0%`

这说明当前注入模型存在明显的生成副作用，优先级高于继续扩大训练。

### 预期改动
- 新增轻量分析脚本，如 `scripts/detect_thinking_leak.py`
- 视情况修改评估脚本，支持泄露统计输出
- `docs/` 中补充止损说明

### 执行动作
1. 建立思考泄露规则检测：
   - `Okay,`
   - `I need to`
   - `the user is`
   - `I should`
   - 其他显式推理痕迹
2. 对当前评估输出批量扫描，统计：
   - 总样本数
   - 泄露样本数
   - 泄露率
3. 输出配置级汇总报告；
4. 增加临时止损策略：
   - 评估时标记泄露样本；
   - 必要时在推理/评估脚本中加入最小过滤或 stop pattern。

### 验收标准
- 能稳定输出每个配置的思考泄露率；
- 评估报告中默认包含泄露统计；
- 泄露止损脚本可重复运行。

### 依赖
- 无

### 优先级
- 最高

## T0.5：建立 baseline 门槛判定规则

### 目标
为是否继续扩大训练或实验提供硬性门槛，避免对负收益模型继续投入。

### 判定规则
任一注入模型若满足以下任一条件，则默认不进入下一轮扩大实验：
1. 平均评分未超过 baseline；
2. 思考泄露率显著高于 baseline；
3. 输出自然度明显下降或出现大量机械化回复。

### 预期改动
- `docs/` 下新增或更新评估规则文档
- 视情况在评估汇总脚本中加入门槛判定字段

### 验收标准
- 每次评估后能明确给出：通过 / 未通过 baseline 门槛；
- “是否继续训练”不再靠主观判断。

### 依赖
- 建议依赖 T0

### 优先级
- 最高

4. 为当前权威口径重新生成泄露统计和门槛判定输入。

### 验收标准
- 当前权威结果目录可追溯；
- 旧目录已归档；
- 报告、门槛判定和结果目录三者一致。

### 依赖

### 优先级

## T1：贯通 gate 配置参数

### 目标
让 YAML 中的 gate 相关超参真正作用到模型实例。

### 问题背景
当前 `gate_max`、`gate_init_bias` 虽存在于训练配置中，但 `src/models/injection.py` 中仍有硬编码路径，导致实验配置不可信。

### 预期改动
- `src/models/persona_steer.py`
- `src/models/injection.py`
- `scripts/train.py`
- 如有需要：`scripts/evaluate.py`

### 执行动作
1. 在 `PersonaSteerConfig` 中增加：
   - `gate_max`
   - `gate_init_bias`
2. 在 `scripts/train.py` 的 `create_model()` 中读取配置并写入模型配置对象；
3. 修改 `SteeringInjection.__init__()`，显式接收：
   - `gate_max`
   - `gate_init_bias`
4. 创建 `DynamicGate` 时传入上述参数，而不是写死。

### 验收标准
- 修改 YAML 后，模型内实际 `gate` 初始化值随之变化；
- 启动日志能打印 `gate_max` 和 `gate_init_bias`；
- 修复前后同输入样本的初始 gate 分布可观察到差异。

### 依赖
- 无

### 优先级
- 最高

## T2：补全 gate 参数可观测性与回归验证

### 目标
确保 gate 的关键行为在日志和测试中可审计、可验证。

### 预期改动
- `src/training/trainer.py`
- `scripts/debug_stage3_gate.py` 或新增轻量验证脚本
- `tests/` 下相关测试文件（如项目允许）

### 执行动作
1. 在模型创建或训练开始时输出：
   - inject_layers
   - gate_hidden_dim
   - gate_max
   - gate_init_bias
2. 在训练日志中补充 gate 统计：
   - mean
   - std
   - min / max
3. 增加最小回归验证：
   - 不同 `gate_init_bias` 是否导致不同初始 gate 均值；
   - `gate_max` 是否真正限制上界。

### 验收标准
- 训练日志可直接看出当前 gate 配置；
- 至少有 1 个回归检查覆盖 gate 参数贯通；
- 调试脚本可复现 gate 分布变化。

### 依赖
- 依赖 T1

### 优先级
- 高

## T3：让 Stage 2/3 的 contrastive loss 真正生效

### 目标
消除“配置里开了对比学习，但训练时 loss 基本为 0”的问题。

### 问题背景
当前很多主配置仍使用 `batch_size: 1`，这会让 `SupervisedContrastiveLoss` 在单样本下直接返回 0。

### 预期改动
- `configs/train_stage2_qwen3.yaml`
- `configs/train_stage3_qwen3.yaml`
- `configs/train_stage3_auto.yaml`
- 其他主线路径配置
- 视情况：`src/training/trainer.py`

### 执行动作
1. 选择主线 Stage 2 / Stage 3 配置；
2. 将主线 `batch_size` 调整到至少 `4`，若显存不足则退到 `2` 并结合其他机制；
3. 在训练日志中打印：
   - `scl_loss`
   - `scl_loss_nonzero_ratio`
   - 有效正样本对数量
4. 明确规定：若 `scl_loss` 长期为 0，则该训练配置视为无效。

### 验收标准
- Stage 3 至少部分 batch 的 `scl_loss > 0`；
- 训练日志中可见正样本对存在；
- `scl_loss` 不再只是名义上配置、实际不参与优化。

### 依赖
- 建议在 T1 后进行

### 优先级
- 最高

## T4：为 contrastive 训练增加成批策略或等效机制

### 目标
提升同批样本中出现正样本对的概率，避免仅靠随机 batch 撞运气。

### 预期改动
- `src/data/aloe_dataset.py`
- 新增 sampler / batch sampler 文件，或在 `scripts/train.py` 中接入
- `src/training/losses.py`
- 相关配置文件

### 执行动作
1. 统计样本中的：
   - `user_id`
   - `personality`
2. 设计最小版 grouped sampler：
   - 优先按 `user_id` 成批；
   - 若不可行，则按 `personality` 成批；
3. 将 sampler 接入 DataLoader；
4. 在训练日志中输出：
   - 当前 batch 的正样本对数量；
   - 正样本构造依据（user_id / personality）。

### 验收标准
- 主线 batch 中正样本对出现频率明显高于随机采样；
- `scl_loss` 非零比例提升；
- 训练过程可解释“为什么此 batch 有对比学习信号”。

### 依赖
- 建议依赖 T3

### 优先级
- 高

## T0.9：baseline vs injected 机制差异归因

- 状态：最高优先级
- 背景：B1 已证明层位有影响，但不足以解释 injected 整体高泄露。
- 目标：解释“为何只要启用注入，模型整体就显著偏离 baseline 的低泄露状态”。
- 子任务：
  - M1：推理链路对齐审计（prompt / template / enable_thinking / decode）
  - M2：planning 外显模式审计（baseline vs injected 文本模式对照）
  - M3：训练目标与输出形式错位审计（是否缺少完成态回答约束）
- 当前判断：层位从主线降级为次级变量，主线切换为机制级差异归因。

- M1 结论：当前 baseline 与 injected 推理链路未对齐。
  - baseline：chat template + `/no_think` + `enable_thinking=False`
  - injected：手写 `用户:...助手:` prompt + 无等价 no_think 控制 + 自定义 decode
  - 判断：应先修推理链路对齐，再继续解释 injected 高泄露根因。

- M1.5：已实现推理链路最小对齐修复。
  - injected 改为使用 chat template
  - injected 显式加入 `/no_think` 与 `enable_thinking=False`
  - injected 改为只解码新增 token
  - 下一步：重跑 baseline + stage3，比较泄露率是否明显下降

- M1.5 结果：stage3 泄露率已从 30.00% 降至 2.22%。
  - stage3 均分从 `2.833` 升至 `3.567`
  - baseline 同轮为 `3.522 / 0.00%`
  - 判断：需用统一链路重跑权威评估，并修正既有结论

- 统一链路权威重跑结果：stage2 当前最佳。
  - baseline：`3.556 / 1.11%`
  - stage1：`3.533 / 0.00%`
  - stage2：`3.589 / 2.22%`
  - stage3：`3.533 / 2.22%`
  - 结论：旧的“baseline 最强、注入高泄露”结论需要整体修正
